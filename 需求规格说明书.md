## 1. 引言

### 1.1 编写目的

本说明书旨在对项目管理系统的功能需求进行结构化描述与规范定义，明确各功能模块的用户故事、验收标准与业务规则，作为后续设计、开发、测试与验收的依据。

### 1.2 项目背景

项目管理系统用于对公司项目全生命周期进行数字化管理，核心目标包括：

- 打通飞书立项/结项流程与项目管理系统之间的数据链路，实现项目信息的自动同步与清洗合并；
- 提供项目清单管理、成员维护、状态审批等核心能力；
- 通过角色化权限体系保障数据安全与操作规范；
- 将项目信息输出至费控、禅道等关联系统，保持全公司项目数据一致。

### 1.3 术语与缩略语

| 术语 | 说明 |
| --- | --- |
| 待合并项目清单 | 飞书同步进入系统的项目暂存区，仅系统管理员可见可操作，用于项目信息清洗与合并 |
| 初始化状态 | 项目同步或创建后的初始状态，尚未进入正式项目清单 |
| 合并项目信息 | 将待合并项目与现有项目进行字段比对、冲突处理后并入项目清单的操作 |
| PMO | 项目管理办公室，负责项目状态变更的审核 |
| 触发式同步 | 由上游事件（如飞书流程完成）触发而非定时调度的数据同步方式 |
| 角色表 | 系统后台维护的项目角色枚举表（如项目负责人、产品经理、开发、测试等），由管理员在「系统管理 → 角色管理」统一增删改，全局共享 |
| 人员属性 | 项目成员的归属类型，固定枚举：国智、合作方、其他 |
| Chip 选择器 | 标签式交互控件，点击切换选中/取消状态，代替原生多选下拉框|

## 2. 总体描述

### 2.1 系统概述

系统围绕"数据同步 → 项目管理 → 审批流转 → 系统管理"四条主线展开，形成从数据接入到业务处理再到治理审计的完整闭环。

### 2.2 用户角色

| 角色 | 权限范围 |
| --- | --- |
| 用户 | 仅具备项目清单与详情的查看权限 |
| 项目负责人 | 具备项目清单与详情的查看、编辑权限 |
| PMO | 具备项目清单与待合并清单的操作权限，以及项目状态审批权限 |
| 系统管理员 | 具备全部系统权限 |

### 2.3 功能模块概览

| 模块 | 功能项数 | 优先级分布 |
| --- | --- | --- |
| 数据同步 | 3 | 高 ×3 |
| 项目管理 | 7 | 高 ×7 |
| 项目审批 | 1 | 中 ×1 |
| 系统管理 | 4 | 中 ×3，低 ×1 |

#### REQ-DS-001 公司人员信息同步

| 项目 | 内容 |
| --- | --- |
| 需求编号 | REQ-DS-001 |
| 所属模块 | 数据同步 |
| 优先级 | 高 |

**用户故事**

作为系统管理员，我希望系统提供公司人员信息的定期同步与手动同步功能，使项目信息中的人员数据能够及时更新，以便在维护项目成员信息时保证数据准确性。

**验收标准**

- **Given**：管理员已登录项目管理后台。
- **When**：管理员点击"人员信息同步"按钮。
- **Then**：系统中人员信息同步为最新数据。

**业务规则**

- 人员信息数据来源于人事系统。
- 定期同步的频率根据系统性能情况确定。

#### REQ-PM-003 成员信息修改

| 项目 | 内容 |
| --- | --- |
| 需求编号 | REQ-PM-003 |
| 所属模块 | 项目管理 |
| 优先级 | 高 |

**用户故事**

作为系统管理员或项目负责人，我希望在项目信息编辑状态下，通过表格形式维护项目成员信息。表格列包含：序号、姓名、角色、人员属性、按项目起止月份动态展开的各月工时列、工时总计。其中角色来源于系统后台维护的角色表（支持多选），人员属性为"国智""合作方""其他"三选一枚举，各月工时按项目周期（开始月份至结束月份）自动展开，工时总计自动汇总。点击「+ 添加成员」可从人事系统人员库中选择人员加入项目表格。

**验收标准**

- **Given**：用户已进入项目信息编辑状态。
- **When**：用户在成员编辑表格中添加/移除成员、调整角色与人员属性、填写各月工时。
- **Then**：表格实时反映成员变动，工时总计自动汇总，项目周期变更后月份列同步调整。

**业务规则**

- 项目成员来源于人事系统同步数据（经去重处理），包含全部相关人员，含项目负责人。
- 角色表（ROLES）由系统管理员在「系统管理 → 角色管理」页面统一维护，增删改即时生效。
- 人员属性固定三个枚举值：国智、合作方、其他。
- 各月工时按项目开始月份至结束月份自动生成列，非必填；项目起止日期变更后月份列随动调整。
- 工时总计列自动计算各月工时之和。
- 角色与人员属性使用 chip 标签式选择器呈现（点击切换选中态），避免原生多选下拉的交互不便。

#### REQ-PM-006 项目新建信息合并

| 项目 | 内容 |
| --- | --- |
| 需求编号 | REQ-PM-006 |
| 所属模块 | 项目管理 |
| 优先级 | 高 |

**用户故事**

作为系统管理员，我希望（针对飞书立项流程）在待合并项目清单中，对该条项目信息点击"合并项目信息"按钮后，系统自动与现有项目进行比对：若存在疑似冲突项目，则展示疑似项目列表供管理员选择；若不存在疑似冲突项目，则进入项目选择界面，由管理员手动选择目标项目或选择"新建合并项目"。系统根据待合并项目与目标项目的信息进行逐项比对，由管理员选择最终保留的数据。确认后，系统完成项目合并，将初始化状态的项目合并至项目清单，并将项目状态自动更新为"进行中"。（选择"新建合并项目"时跳过字段比对步骤。）以便快速识别项目字段冲突，将清洗后的项目字段信息同步到项目清单中。

**验收标准**

- **Given**：管理员进入待合并项目清单中某条项目的详情信息。
- **When**：管理员点击"合并项目信息"按钮，选择潜在冲突项目，逐项选择取信字段（或选择新建合并项目）。
- **Then**：系统识别潜在冲突项目清单，比对两边冲突字段，将项目信息更新或创建到项目清单中，项目状态调整为"进行中"。

**业务规则**

- 通过项目编号、项目名称关键词初步判断潜在冲突项目。
- 本需求针对飞书立项流程。

#### REQ-PM-007 项目结项信息合并

| 项目 | 内容 |
| --- | --- |
| 需求编号 | REQ-PM-007 |
| 所属模块 | 项目管理 |
| 优先级 | 高 |

**用户故事**

作为系统管理员，我希望（针对飞书结项流程）合并流程分四步完成：①管理员首先手动填写结项信息（结项日期、结项原因，均为必填）；②点击"下一步"进入冲突检测，系统自动与现有项目比对（通过项目编号、名称关键词初步判断潜在冲突）；③进入字段逐项比对（结项比对字段多于立项，增加结项日期、结项原因等字段），由管理员逐字段选择保留数据来源；④确认合并，系统将项目状态自动更新为"已结项"并完成信息合并。（结项合并不支持"新建合并项目"选项，必须选择目标项目。）以便在结项环节完成项目信息的清洗与状态流转。

**验收标准**

- **Given**：管理员进入待合并项目清单中某条结项项目的详情信息。
- **When**：管理员依次完成：步骤①填写结项信息（日期+原因）→ 步骤②选择潜在冲突项目（必须选择目标项目，不支持跳过）→ 步骤③逐项选择取信字段 → 步骤④确认合并。
- **Then**：系统完成项目合并，将项目信息更新到项目清单中，项目状态调整为"已结项"。

**业务规则**

- 本需求针对飞书结项流程。
- 结项合并为四步流程（先编辑结项信息 → 冲突检测 → 字段比对 → 确认合并），比立项合并多一步"先手动修改结项信息"。
- 结项信息字段（结项日期、结项原因）为必填项。
- 不支持"新建合并项目"——必须选择现有项目作为合并目标。

### 3.3 项目审批模块

#### REQ-PA-001 项目审批流程

| 项目 | 内容 |
| --- | --- |
| 需求编号 | REQ-PA-001 |
| 所属模块 | 项目审批 |
| 优先级 | 中 |

**用户故事**

作为系统管理员，我希望在项目状态变更时触发项目审批流程，经 PMO 审核通过后完成项目状态调整，以便通过审批机制确保项目状态变更的规范性。

**验收标准**

- **Given**：用户进入项目清单或项目详情界面。
- **When**：用户将项目状态由"进行中"调整为"已完成""挂起"或"已作废"并提交，PMO 角色审核点击"通过"或"拒绝"（拒绝时须填写拒绝原因）。
- **Then**：发起项目审批流程——审核通过则项目状态完成调整；审核拒绝则弹出拒绝原因输入框（必填），提交后其余项目信息仍完成调整，仅项目状态退回原状态，并记录拒绝原因至操作日志。

**业务规则**

- 在项目清单中，项目状态仅支持由"进行中"变更为"已完成""挂起"或"已作废"。
- "已结项"状态由结项信息合并流程触发，不在清单中直接调整。
- 项目信息编辑与项目状态编辑解耦：项目状态调整审批失败不影响人员与字段的调整结果。

#### REQ-SM-004 角色管理

| 项目 | 内容 |
| --- | --- |
| 需求编号 | REQ-SM-004 |
| 所属模块 | 系统管理 |
| 优先级 | 中 |

**用户故事**

作为系统管理员，我希望在后台提供角色管理功能，支持对项目成员可选角色的增删改查维护。角色表为系统全局共享，角色变更即时生效至所有项目的成员编辑界面，以便灵活适配组织架构调整与岗位职责变化。

**验收标准**

- **Given**：管理员已进入后台角色管理页面。
- **When**：管理员添加新角色（输入角色名称后回车或点击添加）、编辑现有角色名称（行内编辑后保存）、删除角色（确认后执行）。
- **Then**：角色表即时更新，所有项目成员编辑界面中的角色选项同步变更。

**业务规则**

- 角色表（ROLES）由系统管理员统一维护，仅「系统管理员」角色可访问角色管理页面。
- 删除角色前需二次确认，并记录操作日志。
- 默认角色不可删除（如"项目负责人"）。
- 角色变更后，已分配给成员的角色不受影响，仅在后续编辑时体现选项变化。
- 角色与人员属性选择器均为 chip 标签式交互（点击切换选中/取消），人员属性枚举固定为：国智、合作方、其他。
